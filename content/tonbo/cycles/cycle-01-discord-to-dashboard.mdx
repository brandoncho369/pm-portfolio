## Summary (BLUF)

Beta testers were abandoning onboarding because Discord, the primary interface,
was unfamiliar and friction-heavy. I moved the interface to a web dashboard and
demoted Discord to an optional notifications channel. Onboarding drop-off fell
from **70% to 16% (21 of 25 new users in the week after launch)**, beating the
≥50%-reduction target.

## Trigger & problem

Pre-launch, onboarding drop-off was ~70%. The activation flow was: pay → receive
onboarding email → join a Discord server → verify email → configure a bot to your
preferences. Each step lost people, and the root cause kept pointing at one thing:
Discord was being asked to be the product's home, and most users didn't want it
there.

## Discovery & user-research synthesis

I gathered structured feedback from **12 beta testers**. **8 of 12** converged on
the same theme — Discord-as-primary-interface — with distinct sub-causes:

- Many didn't have Discord or didn't know how to use it.
- The email-verification step added friction.
- The bot commands were confusing.
- The bot UI and available options felt limited.

This is the load-bearing part of the cycle: a clear majority, one dominant theme,
broken into specific, addressable causes rather than a vague "people didn't like
it."

### 1. Summary (TL;DR)
New users were abandoning onboarding because Discord served as the product's
primary interface, and most users found it unfamiliar and high-friction. This PRD
moves the interface to a web dashboard on the main site (magic-link sign-in) and
demotes Discord to an optional, notifications-only channel. Goal: cut onboarding
drop-off by at least half without losing real-time alert engagement.

### 2. Problem & Background
Onboarding drop-off was ~70%. The flow required users to pay, receive an
onboarding email, join a Discord server, verify their email, and configure a bot.
Beta feedback (below) pointed clearly to Discord-as-interface as the cause: it
asked users to adopt and learn an unfamiliar third-party app just to use the
product they'd paid for.

### 3. Discovery
Structured feedback from 12 beta testers; 8 of 12 independently raised
Discord-as-primary-interface, with these sub-causes:

- No Discord account / unfamiliar with Discord.
- Email-verification friction.
- Confusing bot commands.
- Limited bot UI and options.

### 4. Goals & Non-Goals
**Goals**

- Move the primary interface off Discord and onto a familiar surface (web).
- Preserve real-time alerts (core value).
- Reduce onboarding steps and unfamiliar dependencies.

**Non-Goals**

- A native mobile app.
- Fully removing Discord.
- Replacing the notification mechanism entirely.

### 5. Success Metrics
(Defined before launch.)

- **Primary:** reduce onboarding drop-off by ≥50% within one week. (Result:
  70% → 16%, i.e. 4 of 25 new users — target exceeded.)
- **Guardrail:** real-time alert engagement — share of users who still connect a
  notification channel after Discord became optional.

### 6. Target User
New paying users of Tonbo. A large share are unfamiliar with Discord or unwilling
to adopt it as a primary tool.

### 7. User Stories

- As a new user, I want to use the product from a web dashboard I already know how
  to navigate, so I don't have to learn a new platform to get started.
- As a new user, I want to sign in without a multi-step verification gauntlet, so
  onboarding is fast.
- As an active user, I want real-time alerts, so I don't miss time-sensitive
  signals — but I want connecting that channel to be my choice.

### 8. Solution Overview
The web dashboard becomes the home base. All interactions that previously ran
through Discord bot commands — watchlist adds, the live feed, configuration —
happen in the dashboard. Discord is demoted to a notifications-only channel
(analogous to Telegram alerts) that users opt into for real-time alerts.

New flow: pay → web dashboard on the same site → magic-link sign-in → use
everything → optionally connect Discord for real-time alerts.

Design rationale: Discord had been doing two jobs — interface and notifications —
and was weak at the first but cheap and effective at the second. The two jobs are
split so each runs where it performs best; the unfamiliar surface becomes optional
rather than mandatory.

### 9. Functional Requirements

- Web dashboard hosts all primary interactions (commands, watchlist management,
  live feed).
- Passwordless magic-link authentication.
- Discord connection is optional, used solely to deliver real-time alerts.
- Onboarding requires no third-party app to reach a usable state.

### 10. Edge Cases & Constraints

- User never connects Discord → fully usable via dashboard; alerts surfaced in-app
  where possible.
- Magic-link handling: single-use, time-limited links.
- Constraint: solo operator; favor scope that ships fast over a native app build.

### 11. Risks & Mitigations

- **New attack surface.** Moving the dashboard onto the main site means
  authentication and user data are now directly owned (previously fronted by
  Discord). Mitigations: deliberate passwordless magic-link sign-in (no password
  store); a security audit commissioned from the developer before launch, with
  flagged issues remediated as a precondition of exposing the dashboard.

- **Optional notifications may reduce alert engagement.** Making Discord opt-in
  risks users missing real-time alerts (core value). Mitigation/monitor: track

| --- | --- |

| Swap Discord for another third-party platform as the interface | Same flaw — still forces an unfamiliar third-party app to use the product. |

## Prioritization

I considered four approaches:

- **Native app mirroring the dashboard with push notifications.** Cleanest UX,
  but out of scope for a solo operator on this timeline — too much build for the
  problem.
- **Web-only, no Discord.** Solves the interface problem but breaks
  notifications: I'd either spam users with email or lose real-time alerts
  entirely — and real-time alerts are core value.
- **Swap Discord for another third-party platform (e.g. Telegram) as the primary
  interface.** Same fundamental flaw — still forcing users onto an unfamiliar
  third-party app to *use the product*.
- **Hybrid (chosen).** The web dashboard becomes the home base — all commands,
  watchlist adds, and the live feed live there. Discord is demoted to
  notifications only, like a Telegram alert channel.

**The insight:** Discord was quietly doing *two* jobs — interface and
notification channel — and it was bad at the first but cheap and good at the
second. I unbundled them and put each job where it's strongest: interface on the
web, notifications on Discord. That keeps the core value (real-time alerts) while
making the unfamiliar surface *optional* instead of *mandatory*.

## Build & sprint summary

New flow: **pay → web dashboard on the same site → magic-link sign-in → do
everything from the dashboard → optionally connect Discord for real-time alerts.**
Discord went from a required gate to an opt-in add-on.

Relocating the dashboard onto the main site introduced a new attack surface: for
the first time I directly owned **authentication and user data** (previously
fronted by Discord). Two responses:

- I deliberately chose **passwordless magic-link sign-in** before launch — no
  password store, so there's nothing to breach on that front.
- I'm not a security specialist, so I **commissioned a security audit from my
  developer before exposing the dashboard** and had the flagged issues remediated
  as a precondition of launch.

### Retro & lessons

- The reusable lesson is the **unbundling move**: when one tool is silently doing
  two jobs, split it so each lives where it's strongest.
- **Honest note on scale:** these are small, different cohorts (12 beta testers,
  25 post-launch onboarders). The result is dramatic but directional, not
  statistically powered.
- **Honest note on security:** I made the right structural calls — passwordless
  auth and a pre-launch audit gating the new surface — but I was dependent on my
  developer to know *what* to worry about. The growth edge is building my own
  baseline understanding of the surface so I can scope the risk myself, not just
  commission someone to find it.
