## Summary (BLUF)

Testers couldn't track watches outside Tonbo's catalog — their only option was to
email me to add each model by hand. I shipped a self-serve flow that resolves any
off-catalog model in under 60 seconds, for under 1¢ per model and zero operator
effort. Model-add support emails dropped 87% within 7 days, and every self-serve
attempt succeeded within two tries.

## Trigger & problem

During tester feedback, one complaint recurred: the specific watch a collector
wanted to track wasn't in the catalog. The only fallback was a support email,
after which I, the sole operator, manually added the model. That was friction for
the user (wait on a human, every single time) and a hard bottleneck for me (every
catalog gap routed through one person).

## Discovery & user-research synthesis

The signal came straight from tester feedback and from the volume of "please add
this model" support emails, which doubled as a clean frequency measure of how
often the gap bit.

The insight that shaped the solution: **each "it's not on here" email is a
precise demand signal.** It tells me exactly which long-tail reference a real user
wants. The problem and the input to the solution are the same event, so the right
design treats the complaint as the trigger for supply, not as a ticket to triage.

## The PRD

<PRD>

### Problem
Users can't add watches outside the existing catalog without a manual support
request.

### Target user
Japanese watch collectors tracking specific reference models, many of them in the
long tail outside the core catalog.

### Success metrics (defined upfront)
- **Primary:** reduce model-add support email volume by ≥75% within 7 days of
  launch.
- **Guardrail:** model-data integrity — no duplicate or non-existent models
  entering the database via self-serve adds.

### In scope
- A self-serve path to resolve and add any single off-catalog model.

### Out of scope
- Pre-loading the full ~10k+ universe of watches (cost, database bloat, and the
  majority never demanded).

### User stories
- As a collector, I want to add a watch that isn't in the catalog without
  emailing support, so I can build my watchlist immediately.

### Edge cases
- Model doesn't exist → return "model not found."
- User enters a nickname instead of a reference → prompt for the exact model
  number.
- Cost / abuse → rate limiting on requests.

### Constraints
- 2–3 day build; solo operator (no communication delay, so iteration is fast).

</PRD>

## Prioritization

I considered four options:

- **A — Add to watchlist now, hook up data later by hand.** Unblocks the user
  instantly, but the manual step (adding the model to the scraper regex pattern
  and to comp-data collection) still routes through me. The bottleneck is
  deferred, not removed.
- **B — Batch the top-requested models weekly.** Neglects the long tail entirely,
  and a user who requests early in the week waits days. Still manual
  prioritization on my end.
- **C — Just add all 10k+ models.** Scraping credits skyrocket, the database
  bloats, and most of those watches are never demanded — paying to supply
  inventory nobody asked for.
- **D — On-demand resolution (chosen).** A sticky input at the bottom of the
  existing model dropdown takes the user's reference number → an API call to
  Claude maps it to the correct brand and family → the model is written to the
  scraping database → the same weekly comp-detection/calculation job is run
  on-demand for just that one model. The user gets comp data in under 60 seconds
  and adds the watch from the same menu. Under 1¢ per model, zero operator effort.

**Why D wins:** it's instant for the user (like A, unlike B), requires zero manual
operator work (unlike A), and scales cost to actual demand (unlike C). The LLM is
the piece that collapses the one step — brand/family disambiguation — that
previously required a human. In effect, D is "Option A with the hook-up step
automated out of existence."

## Build & sprint summary

Shipped in **2 days**. The request input lives at the bottom of the existing
model-selection dropdown (sticky), deliberately on the same surface where users
add a watch to their watchlist — so resolving a missing model and completing the
actual job happen in one place, with no separate flow to drop out of.

Pipeline: user enters reference number → Claude matches brand/family → insert into
scraping DB → one-off run of the existing comp-detection/calculation job → result
surfaced to the user → user adds to watchlist inline.

*No scope changes.*

**Completion:** 100% of self-serve attempts succeeded within two tries. The
retries were users who entered a nickname first and then the exact reference
number after the "not found" prompt — i.e., the edge-case handling worked as
designed: the error message taught users to self-correct rather than giving up.

**Conversion:** 100% of users who resolved a model added it to their watchlist
immediately, because it happens in the same dropdown — the design collapses
resolution and the actual job into a single step, so there's no funnel drop-off.

**Guardrail — data integrity:** a weekly validation cron checks for duplicate and
non-existent models, acting as the safety net on anything the LLM matcher gets
wrong.

### Retro & lessons

The guardrail is currently *lagging* — it runs weekly, not at point of insertion.
A confidently-wrong match (a valid-looking reference mapped to the wrong family)
could feed bad comp data for up to a week before the cron catches it. Because comp
accuracy is Tonbo's core value, the next iteration is a real-time confidence check
or match-confirmation step at insertion — consistent with how I already
communicate uncertainty elsewhere in Tonbo (confidence levels on sparse-data
alerts).
